# Lumen — architecture de reference

> Ce document capture l'etat de la sous-app Lumen (cours pedagogiques) apres
> les iterations v2.42 → v2.49 (pivot GitHub, auto-manifest, scaffold, focus
> mode, recherche fulltext FTS5, hardening + simplification UX). Destine aux
> futures sessions de dev (humaines ou AI) pour eviter de re-explorer le code.

## Mental model

```text
Promo (Cursus)
  └─ 1 Organisation GitHub  (promotions.github_org)
       └─ N Repositories
            ├─ cursus.yaml  (manifest OPTIONNEL — auto-manifest sinon)
            │   ├─ project: "Titre d'affichage du cours"         (required, string)
            │   ├─ cursusProject: "Nom d'un projet Cursus"       (optional, lien repo<->projet)
            │   ├─ chapters: [{title, path, duration?, summary?, prerequis?, section?}]
            │   ├─ resources: [{path, kind, title}]              (optional)
            │   ├─ autoGenerated: bool                            (true si genere depuis l'arbre)
            │   └─ truncated: bool                                (true si arbre GitHub > 100k entrees)
            └─ N fichiers .md referencs par chapters[].path
```

**Lumen n'est pas un CMS** : le prof edite ses cours en dehors de l'app
(VS Code, github.com, Copilot) puis sync. Lumen est une **liseuse avec
integration profonde dans le parcours Cursus** — le levier de valeur est
l'integration (devoirs, projets, recherche), pas la qualite visuelle du
rendu. Direction strategique v2.48 : **l'app s'efface derriere le contenu**
(focus mode default ON, sidebar Cursus globale masquee, sidebar Lumen flat,
auto-resume du dernier chapitre, suppression de l'accuse de lecture).

## Schema DB (migrations v56 → v59)

| Table | Cle | Role |
|---|---|---|
| `lumen_github_auth` | (user_type, user_id) | Token PAT par user, **chiffre AES-GCM** (format `enc:...`). Migration silencieuse au boot serveur via `migrateLumenTokensAtBoot()` (v2.47). |
| `lumen_repos` | id, UNIQUE(promo_id, owner, repo) | Un repo par promo. Contient `manifest_json` parse + `manifest_error` + `project_id` + `last_commit_sha` + `last_synced_at` + **`is_visible`** (v58, default 0 nouveaux repos, default 1 repos pre-existants). |
| `lumen_file_cache` | (repo_id, path) | Cache markdown avec images inlinees (data URI). Pruned au sync + TTL 30j. |
| `lumen_chapter_notes` | (student_id, repo_id, path) | Notes privees etudiant, max 10k chars. |
| `lumen_chapter_reads` | (student_id, repo_id, path) | **Deprecated v2.48** : tracking de lecture supprime cote UX. Table conservee pour rollback eventuel, plus de write. |
| `lumen_chapter_travaux` | (travail_id, repo_id, chapter_path) | Liaison N:M devoir <-> chapitre (v57). Cascading deletes. |
| `lumen_chapter_fts` | (repo_id, chapter_path) virtual FTS5 | **Index fulltext** (v59) avec `tokenize='unicode61 remove_diacritics 2'`. Alimentee lazily au fetch chapitre. |

### Promo ↔ Organisation GitHub

Colonne `promotions.github_org TEXT` (v56). Un prof admin de la promo la
configure via le modal settings dans `LumenView`. Le sync liste tous les
repos non-archives de l'org via octokit.

### Repo ↔ Projet Cursus

Colonne `lumen_repos.project_id INTEGER REFERENCES projects(id)`. Deux voies :

1. **Yaml maitre** : `cursusProject: "API Meteo"` dans `cursus.yaml`. Resolu
   au sync par `findProjectByNormalizedName(promoId, name)` (trim + lowercase
   + Unicode NFD).
2. **UI fallback** : teacher admin assigne manuellement via `LumenLinkRepoModal`.
   **Refuse 409 si le manifest declare deja un cursusProject** (yaml prioritaire).

Ambiguite (plusieurs projets meme nom normalise) : refus explicite avec
`manifest_error`. `projects.name` n'est pas UNIQUE en DB — gere cote app.

### Visibilite repo (v58)

`lumen_repos.is_visible INTEGER NOT NULL DEFAULT 0` :

- **Default 0 (masque)** pour les nouveaux repos sync apres v58. Le prof doit
  cliquer l'icone oeil dans la sidebar pour publier explicitement.
- **Migration v58 a marque `is_visible=1`** pour tous les repos deja en DB
  au moment de l'upgrade, pour ne pas casser l'experience etudiante.
- Les students ne voient JAMAIS les repos masques (filter applique sur 4
  endpoints : `/repos/promo/:id`, `/repos/:id`, `/repos/:id/content`,
  `/repos/by-project-name`, `/promos/:id/search`).
- Teacher / admin / TA voient toujours tout.

### Index fulltext FTS5 (v59)

```sql
CREATE VIRTUAL TABLE lumen_chapter_fts USING fts5(
  repo_id UNINDEXED,
  chapter_path UNINDEXED,
  title,
  content,
  tokenize='unicode61 remove_diacritics 2'
);
```

- **Indexation lazy** : declenchee dans `fetchChapterContent` au cache miss
  uniquement. Un chapitre jamais ouvert n'est pas searchable jusqu'a sa
  premiere lecture par n'importe quel user. Trade-off accepte : pre-fetch
  tous les chapitres au sync coute trop d'API GitHub.
- **Plain text extraction** via `markdownToPlainText()` : strip code blocks /
  HTML / images / liens / formatting markers. Cap a 200KB en input pour
  eviter le ReDoS sur markdown bombs.
- **Snippet XSS-safe** : strategie sentinel avec `\u0001LUMENMARK\u0001` /
  `\u0002LUMENMARK\u0002`, escape HTML cote serveur, puis substitution par
  `<mark>...</mark>`. Le client peut afficher en `v-html` sans risque.
- **Prune coordonne** : `pruneLumenChapterFtsForRepo` appele alongside
  `pruneLumenFileCacheForRepo` au sync pour garder l'index coherent.

### Design system

- **Editorial content-first** : Newsreader serif pour le markdown body, sans-serif pour le chrome
- **Type scale fort** h1 2.2em → h6 0.9em avec letter-spacing tight
- **Code blocks** : wrapper `.lumen-codeblock` avec badge langue + atom-one-dark + JetBrains Mono
- **Math KaTeX** : display + inline avec cache global (extraction pre-marked, reinjection post-DOMPurify)
- **Schemas Mermaid** : code fence ```mermaid -> SVG via mermaid.render dynamique, theme dark
- **Admonitions Obsidian** `[!NOTE]` `[!WARNING]` etc. avec icones SVG lucide
- **Copy button** injecte dynamiquement sur `pre.lumen-code` post-render
- **Hint blocks** accent-softened pour les sections "Devoirs lies" et "Chapitres a revoir"

## Decisions importantes a connaitre

1. **Octokit v22 est ESM-only** — `server/services/githubClient.js` utilise
   `dynamic import()` cache. Ne pas tenter `require('@octokit/rest')`.

2. **Projets Cursus par nom, pas par ID** cote frontend — legacy
   `travaux.category = projects.name` utilise partout. La navigation projet
   = `appStore.activeProject = name; router.push('/devoirs')`.
   **Pas de route /projects/:id.**

3. **Manifest toujours maitre** sur `cursusProject` :
   - Champ present → ecrit `project_id` au sync (NULL si ne resout pas).
   - Champ absent → preserve la valeur existante (UI fallback).
   - L'UI fallback refuse 409 si le manifest declare deja le champ.

4. **`project` (manifest) vs `cursusProject` (manifest)** sont DEUX champs distincts :
   - `project` = titre d'affichage du cours (required, historique)
   - `cursusProject` = nom du projet Cursus auquel lier (optional, ajoute v2.41)

5. **Auto-manifest est un fallback** : si `cursus.yaml` est absent, le sync
   appelle `generateAutoManifest(octokit, ...)` qui fait 1 appel `git.getTree`
   recursive et derive un manifest depuis la convention de nommage (README →
   "Accueil", dossiers top-level → sections humanisees, fichiers `.md` tries
   par prefixe numerique). Le manifest est marque `autoGenerated: true` et
   l'UI affiche un badge Sparkles. Le prof peut toujours ajouter un
   `cursus.yaml` qui prendra le pas au prochain sync.

6. **Scaffold "Nouveau cours"** : `createRepoWithScaffold(octokit, ...)` cree
   un repo prive via `createInOrg(auto_init: true)`, pousse les fichiers du
   scaffold (README + projet.md + process-daily.md + prosits/1-exemple.md +
   .gitkeep dossiers) en **1 commit initial** via Trees API (createBlob
   parallel → createTree → createCommit → updateRef). Retry exponentiel sur
   `getBranch` pour absorber la latence post-auto_init. Le nouveau repo
   herite `is_visible=0` (masque par defaut).

7. **Images dans les cours privees** : `inlineImages` parallelise 5 fetches
   en parallele puis applique le budget total sequentiellement. Limites :
   1.5 MB/image, 6 MB/chapitre, placeholder si depasse. `<img>` cote
   renderer n'a pas besoin d'auth (data URIs).

8. **Tokens GitHub chiffres** avec `server/utils/crypto.js` (AES-256-GCM
   derive JWT_SECRET via pbkdf2). Format DB : `enc:<base64>`. **Migration
   silencieuse au boot** via `migrateLumenTokensAtBoot()` appele dans
   `server/index.js` apres `queries.init()`. Tests : `process.env.JWT_SECRET`
   en tete de fichier.

9. **Le sync est idempotent + parallelise** par batch de 5 repos.
   `syncPromoRepos` snapshot les IDs deja en DB AVANT le sync pour eviter
   le race "Nouveau cours" cree pendant le sync se fait pruner.

10. **Mode focus par defaut ON** : `useLumenFocus` retourne `true` si la pref
    n'est jamais explicitement set. Avec le bouton "Cursus" home dans la
    topbar Lumen, l'utilisateur a toujours une issue claire pour ressortir.

11. **Recherche FTS5 lazy** : un chapitre n'est indexe qu'au premier fetch.
    Les students cherchant un cours pas encore lu n'auront rien — c'est un
    trade-off accepte (alternative = pre-fetch tous les chapitres au sync,
    cout API GitHub trop eleve).

## Flow utilisateur complet (reference)

### Premier setup d'une promo (admin)

1. Admin clique "Configurer organisation" dans LumenView (modal settings)
2. Entree du nom de l'org GitHub. PUT `/api/lumen/promos/:id/github-org`.
3. Admin clique "Synchroniser" → `syncPromoRepos` liste tous les repos
   non-archives + parse `cursus.yaml` (ou auto-manifest si absent)
4. Pour chaque repo : upsert dans `lumen_repos`, manifest_json mis a jour,
   `is_visible=0` par defaut (nouveaux), repos absents de GitHub pruned
   (mais nouveaux repos crees pendant le sync preserves via snapshot)
5. Admin parcourt la sidebar et clique l'icone oeil sur les repos a publier
   pour les rendre visibles aux students

### Admin cree un nouveau cours

1. Bouton "Nouveau cours" dans la topbar Lumen (teacher/admin)
2. Modal : slug du repo + titre du bloc
3. POST `/api/lumen/promos/:id/repos` → `createRepoWithScaffold` cree le
   repo prive sur GitHub + pousse le scaffold en 1 commit
4. Sync auto post-creation pour ramener le repo dans la sidebar
5. Repo cree avec `is_visible=0` (admin doit cliquer l'oeil pour publier)
6. Admin edite le contenu sur GitHub / VS Code, resync depuis Lumen quand
   prêt

### Etudiant ouvre Lumen

1. Pas de token → LumenGithubConnect (PAT paste)
2. Token ok → fetch repos visibles de la promo → sidebar remplie
3. **Auto-resume** : `useLumenLastChapter.get(promoId)` → ouverture directe
   du dernier chapitre lu (zero clic). Deep-link URL prioritaire.
4. **Mode focus actif** : sidebar Cursus globale masquee. Bouton "Cursus"
   en haut a gauche pour ressortir au dashboard.
5. Si stale > 15 min, auto-sync declenche

### Etudiant cherche un mot-cle

1. Tape dans le filter input de la sidebar
2. Debounce 300ms, appel `searchChapters(promoId, q)` → endpoint FTS5
3. Filter scoping = repos visibles uniquement
4. Resultats avec snippets `<mark>` highlightes affiches au-dessus de la
   liste repos. Click = ouvre le chapitre dans le viewer (auto-resume
   memorise la nouvelle position pour la prochaine session)

### Etudiant ouvre un projet dans /devoirs

1. `appStore.activeProject = nom` + navigation
2. `StudentProjetFiche.vue` se monte avec `projectKey: nom + promoId`
3. `<LumenProjectSection>` fetch repos lies via `/repos/by-project-name`
4. Cache entierement si 0 repos (student), liste repos + chapitres si > 0

### Prof lit un chapitre et veut le lier a un DS

1. Ouvre le chapitre dans LumenChapterViewer
2. Footer "Devoirs lies" affiche bouton "+ Lier un devoir" (teacher only)
3. LumenLinkDevoirModal liste les travaux de la promo via `getGanttData`
4. Toggle click-to-link, feedback immediat
5. Au retour dans le viewer, la liste des devoirs lies est refresh

## Non-scope explicite (a NE PAS implementer sans nouvelle deep interview)

- Co-edition temps reel des cours
- Historique git visible dans l'UI
- Formats non-markdown (PDF, docx, notebooks) cote viewer
- Migration UNIQUE(promo_id, name) sur projects (risque donnees)
- OAuth device flow (PAT uniquement)
- Re-implementation de l'accuse de lecture (decision tranchee v2.48 :
  feature retiree car silencieuse, peu fiable, et inconfortable UX)
- Pre-indexation eager FTS5 au sync (cout API GitHub vs gain marginal)
- Editeur in-app pour ecrire le markdown (le prof reste sur VS Code/GitHub)
